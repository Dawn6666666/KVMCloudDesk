本文件描述 **Windows 开发端** 与 **CentOS 部署端** 的环境差异和共同约定，确保编程 agent：

## 3. Windows 开发环境说明

### 3.1 Windows 的任务

- 编写 Java Swing 客户端 GUI
- 开发 Spring Boot 后端（mock 模式）
- 前后端都在 Windows 本地运行和调试
- 不需要 KVM、libvirt、QEMU

### 3.2 Windows 上绝对不允许做的事情

- [禁止] 不要加载 `/usr/lib64/libvirt.so.0`（这是 Linux .so 文件）
- [禁止] 不要在 Windows 上使用 `@Profile("libvirt")` 启动应用
- [禁止] 不要在业务代码中调用 `virsh`
- [禁止] 不要使用 `Runtime.exec()` 执行任何 virsh 命令
- [禁止] Swing 客户端代码不要直接依赖任何 libvirt API

### 3.3 Windows 开发时的 mock 数据建议

Mock 模式应模拟以下真实环境数据：

```
宿主机: centos, CPU: AMD Ryzen 5 3600, 4 核, 7.5 GiB
虚拟机: demo (1 GiB RAM, 2 vCPU, 已关闭)
网络: default (NAT, 192.168.122.0/24, 活动)
存储池: default (/var/lib/libvirt/images, 活动)
镜像: ubuntu16.04.7.img (10 GB)
```

### 3.4 Spring Boot Profile 约定

```
application-mock.yml      → Windows 开发时激活
application-libvirt.yml   → CentOS 部署时激活
```

## 12. 对编程 agent 的关键提醒

### 12.1 环境隔离

1. **Windows 上只开发和测试 mock 模式**，不要加载 libvirt.so。
2. Swing 客户端不直接依赖 libvirt，只通过 HTTP API 调用后端。
3. 后端必须支持 `mock` 和 `libvirt` 两个 Spring Profile。
4. `mock` profile 在 Windows 上可完整运行（前后端均本地）。
5. `libvirt` profile 只在 CentOS 上运行。

### 12.2 Bean 隔离要求

```
@Profile("libvirt")  → LibvirtConnect, LibvirtHostService, LibvirtVMService, ...
@Profile("mock")     → MockConnect, MockHostService, MockVMService, ...
```

所有 libvirt 相关 Bean 必须用 `@Profile("libvirt")` 隔离，确保 Windows 启动时不会加载。

## 13. 不允许事项

### 13.1 禁止 virsh 调用

以下调用方式在**项目业务代码中一律禁止**：

```java
// [禁止] 禁止
Runtime.getRuntime().exec("virsh list --all");
ProcessBuilder pb = new ProcessBuilder("virsh", "start", "vm-name");
```

```python
# [禁止] 禁止
subprocess.run(["virsh", "list"])
os.system("virsh start demo")
```

### 13.2 virsh 唯一合法用途

- SSH 到 CentOS 后人工执行，用于环境诊断和 API 行为验证
- 例如：`virsh list --all` 确认某个 VM 是否已被 libvirt API 正确操作

### 13.3 其他禁止事项

- [禁止] Windows 上加载 `/usr/lib64/libvirt.so.0`
- [禁止] Windows 上启动 `@Profile("libvirt")` 应用
- [禁止] Swing 客户端直接 import libvirt 相关包
- [禁止] 硬编码 `/usr/lib64/libvirt.so.0` 路径（应通过配置文件注入）

## 14. 建议的验证命令
