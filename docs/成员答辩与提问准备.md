### 1. 提问：“后端是如何区分本地 Windows 模拟开发与 CentOS 真实 KVM 部署环境的？”
*   **对应代码文件**：
    *   [LibvirtConnectionManager.java](file:///g:/Code/Other/KVMCloudDesk/backend/src/main/java/com/example/kvm/backend/service/libvirt/LibvirtConnectionManager.java) (第 12 行)
    *   [LibvirtVmService.java](file:///g:/Code/Other/KVMCloudDesk/backend/src/main/java/com/example/kvm/backend/service/libvirt/LibvirtVmService.java) (第 24 行)
*   **原理解析**：
    后端通过 `@Profile` 注解实现多环境装配。`LibvirtConnectionManager` 与 `LibvirtVmService` 上标记了 `@Profile("libvirt")`。Windows 环境激活 `mock` profile，不加载 libvirt 库，避免报错；CentOS 环境下激活 `libvirt` profile 才会加载底层动态链接库。

### 2. 提问：“既然是通过 JNA 调用 C API，底层内存是由 C 语言库在物理堆外分配的。JVM 的 GC（垃圾回收）管不到堆外内存，你是如何防止内存泄漏的？”
*   **对应代码文件**：
    *   [LibvirtVmService.java](file:///g:/Code/Other/KVMCloudDesk/backend/src/main/java/com/example/kvm/backend/service/libvirt/LibvirtVmService.java) (第 36-59 行 `listVms` 方法)
    *   [LibvirtConnectionManager.java](file:///g:/Code/Other/KVMCloudDesk/backend/src/main/java/com/example/kvm/backend/service/libvirt/LibvirtConnectionManager.java) (第 14-16 行，第 51-55 行 `free` 方法)
*   **原理解析**：
    JNA 会在堆外分配物理内存。
    - **对象释放**：在 `listVms` 方法中获取的 domain 指针，使用完毕后在 `finally` 块中调用 `virDomainFree` 释放。
    - **数组释放**：通过 `virConnectListAllDomains` 获取的指针数组由底层分配，在 `finally` 块中需要通过自定义加载的 `libc.free` 方法释放堆外内存。

### 3. 提问：“虚拟机创建的代码是如何实现的？是如何将硬件参数注入 KVM 宿主机的？”
*   **对应代码文件**：
    *   [LibvirtVmService.java](file:///g:/Code/Other/KVMCloudDesk/backend/src/main/java/com/example/kvm/backend/service/libvirt/LibvirtVmService.java) (第 67-183 行 `createVm` 方法)
*   **原理解析**：
    1. **名冲突校验**：通过 `virDomainLookupByName` 判断同名虚拟机是否已存在。
    2. **磁盘文件克隆**：判断镜像格式，在 images 目录下复制文件作为虚拟机的系统盘。
    3. **动态 XML 装配**：根据入参拼接虚拟机 XML 配置文件，将 CPU 核心数、内存大小和网桥名称写入对应节点。
    4. **虚拟机注册定义**：调用 `virDomainDefineXML` 接口注册虚拟机。

### 4. 提问：“前端 noVNC 控制台可以直接操控虚拟机桌面，后端是如何扮演数据流代理角色的？底层是基于什么网络协议？”
*   **对应代码文件**：
    *   [VncProxyWebSocketHandler.java](file:///g:/Code/Other/KVMCloudDesk/backend/src/main/java/com/example/kvm/backend/config/VncProxyWebSocketHandler.java) (第 29-89 行 `afterConnectionEstablished`，第 92-107 行 `handleBinaryMessage`)
*   **原理解析**：
    前端 noVNC 客户端通过 WebSocket 与后端连接，而虚拟机控制台监听的是原生 TCP 端口。
    1. **端口获取**：通过 `vmService.getVm` 拿到虚拟机的 VNC 物理端口。
    2. **TCP 桥接**：后端通过 `new Socket("localhost", vncPort)` 建立与该端口的 TCP 连接。
    3. **双向转发**：
       - **后端到前端**：启动 `vnc-reader` 异步线程读取 TCP Socket 字节，通过 WebSocket 的 `BinaryMessage` 发送给前端。
       - **前端到后端**：在 `handleBinaryMessage` 中拦截前端传回的键鼠操作，写入 TCP Socket 传给虚拟机。

*   **对应代码说明**：
    *   结合 Axios 拦截器与 Pinia Store （答辩时指出 [DashboardView.vue](file:///g:/Code/Other/KVMCloudDesk/client-web/src/views/DashboardView.vue) 依赖的全局 log 逻辑）
