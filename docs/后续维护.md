## 当前项目状态

项目为多模块结构，各模块职责如下：

- `common`：提供 DTO、请求对象以及统一响应结构 `ApiResponse<T>`。
- `backend`：提供 REST API（22 个端点），支持 mock 和 libvirt 双运行模式。在 libvirt 模式下，通过 JNA 映射约 57 个 libvirt C 函数和 4 个 Structure，连接 `/usr/lib64/libvirt.so.0` 和 `qemu:///system`。
- `client-swing`：基于 Swing 编写的桌面客户端（1180×760，FlatLaf 明暗主题，MigLayout 布局，SwingWorker 异步）。
- `client-web`：基于 Vue 3 + TypeScript + Element Plus + Pinia + ECharts 编写的网页客户端。主色调为 `#ca6a1f`，提供内存使用率、虚拟机状态与存储容量图表，以及全局操作日志追踪面板和关机轮询进度指示。

已在 CentOS 真实环境中实测通过的接口与功能：

- 宿主机系统信息：优化了获取主机名逻辑，移除了导致 15 秒反向解析超时的原生 `virConnectGetHostname` 调用，改为依次读取 `$HOSTNAME`、`/proc/sys/kernel/hostname`、`/etc/hostname` 及 `InetAddress` 进行多级回退。
- 虚拟机列表及详情查询。
- 虚拟机创建：支持在 libvirt 环境下基于系统镜像或光盘进行创建，自动生成包含 virtio 设备、VNC 图形、host-passthrough CPU 以及 ACPI/APIC 特性的 domain XML。
- 虚拟机删除：支持取消虚拟机定义并物理删除关联的磁盘映像文件。
- 虚拟机控制操作：包括启动、关机、重启、强制关闭、暂停、恢复。
- 虚拟网络查询与开关：包括启动和停止默认网络。
- 存储池与存储卷的列表查询。
- 快照生命周期管理：包括快照创建、列表查询、恢复与删除，支持通过 virDomainSnapshotCurrent 调用识别并高亮显示当前的活跃快照，已通过 qcow2 虚拟机测试验证。

已知开发约束：

- 在 Windows 本地开发时，不要启动 libvirt 配置，只能运行 mock 配置。
- 虚拟机控制中的关机操作依赖虚拟机内部系统对 ACPI 信号的响应，若需要立即断电请使用强制关闭接口（对应调用 `virDomainDestroy`）。
- 业务代码中严禁通过 `Runtime.getRuntime().exec()`、`ProcessBuilder` 或其他任何方式调用 virsh 命令行。
- `CreateVmRequest.diskSizeGb` 在 mock 模式下生效，但 libvirt 模式的 `createVm()` 未实际执行磁盘 resize（磁盘大小取决于模板镜像文件）。

## 架构关键设计

### 资源管理模式

所有 libvirt 域操作通过 `withDomain` 模板方法完成连接管理和资源释放，该模式同样适用于 `withNetwork`、`withPool` 与 `withSnapshot`。`LibvirtConnectionManager.open()` 每次新建连接，无连接池。

### 状态映射

libvirt 7 种 domain 状态在代码中做了合并映射：
- state 1 (RUNNING) + 2 (BLOCKED) → "运行"
- state 3 (PAUSED) → "暂停"
- state 4 (SHUTDOWN) + 5 (SHUTOFF) → "关闭"
- state 6 (CRASHED) → "异常"

### 安全措施

`LibvirtUtil.xml()` 解析 libvirt XML 时启用 XXE 防护（禁用 DOCTYPE、关闭实体引用展开）。

### 关机轮询

Web 客户端关机后每 1.5 秒轮询虚拟机状态，最多 15 次（共计约 22.5 秒），进度条渐近递增，超时后显示强制断电按钮。

## 常见问题排查

- **虚拟机启动或关机延迟问题**：
  若 `virsh start` 启动虚拟机或关机操作响应缓慢，通常原因为 CentOS 无法解析自身主机名 `centos` 导致域名解析超时。
  必须保证 CentOS 虚拟机的 `/etc/hosts` 文件末尾包含了主机名的回环解析映射：
  ```text
  127.0.0.1 centos
  ::1 centos
  ```

## 常用命令与联调指南

### 后端 mock 运行模式

```bash
java -jar backend/target/kvm-cloud-backend.jar --spring.profiles.active=mock
```

### CentOS 真实 libvirt 后端打包与部署

1. **打包后端（指定 JDK 21）**：
   在 Windows 开发机器的 PowerShell 中执行以下命令（通过绝对路径运行 Maven 并临时指定 Java 21 环境变量）：
   ```powershell
   powershell -Command '$env:JAVA_HOME = "C:\Users\24831\.jdks\ms-21.0.10"; & "D:\Jetbrains\IntelliJ IDEA 2025.2.4\plugins\maven\lib\maven3\bin\mvn.cmd" clean package -DskipTests'
   ```
2. **前后端合并打包与资源同步**：
   如果修改了前端网页客户端，需要重新编译并放入后端的静态资源托管目录下：
   ```powershell
   # 1. 编译前端静态资源
   cd client-web
   npm run build
   cd ..
   
   # 2. 清空并拷贝静态资源至后端 static 目录
   Remove-Item -Path 'D:\Code\Other\kvm\backend\src\main\resources\static\*' -Recurse -Force
   Copy-Item -Path 'D:\Code\Other\kvm\client-web\dist\*' -Destination 'D:\Code\Other\kvm\backend\src\main\resources\static' -Recurse -Force
   
   # 3. 运行上方步骤 1 中的编译后端命令重新打包
   ```
3. **上传并覆盖部署至 CentOS**：
   ```bash
   # 上传 jar 包
   scp backend/target/kvm-cloud-backend.jar centos:/tmp/kvm-cloud-backend.jar
   
   # 登录 CentOS 重启服务
   ssh centos "pkill -f kvm-cloud-backend.jar || true"
   ssh centos "nohup java -jar /tmp/kvm-cloud-backend.jar --spring.profiles.active=libvirt > /tmp/kvm.log 2>&1 < /dev/null &"
   ```
